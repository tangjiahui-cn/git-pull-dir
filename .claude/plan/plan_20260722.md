## Phase 2：核心源码实现

### 2.1 文件创建顺序（按依赖关系）

```
src/
├── errors.ts      # 无依赖，最先实现
├── utils.ts       # 依赖 errors.ts
├── cli.ts         # 依赖 errors.ts, utils.ts
├── clone.ts       # 依赖 errors.ts, utils.ts
└── index.ts       # 依赖 cli.ts, clone.ts, utils.ts
```

### 2.2 `src/errors.ts` — 自定义错误类型

**职责：** 定义工具专属错误类型，统一错误消息格式。

| 错误类 | 退出码 | 触发条件 |
|--------|--------|----------|
| `GitPullDirError`（基类） | 1 | 所有可预见的工具级错误 |
| `GitNotInstalledError` | 1 | Git 未安装或版本 < 2.25 |
| `InvalidArgumentError` | 1 | 参数缺失或格式错误 |
| `RemoteUnreachableError` | 1 | 远程仓库不可达 |
| `DirNotFoundError` | 1 | 指定目录在仓库中不存在 |
| `DirExistsError` | 0 | 本地目录已存在（非中断错误，交互处理后退出码仍为 0） |
| `TimeoutError` | 1 | 整体流程超过 3 分钟 |
| `DiskSpaceError` | 1 | 磁盘空间不足 |
| `CancelError` | 130 | 用户 Ctrl+C 中断 |

**→ 提交：** `git commit -m "add errors.ts"`

### 2.3 `src/utils.ts` — 工具函数

**职责：** 通用的辅助函数，不涉及 Git 操作。

| 函数 | 签名 | 说明 |
|------|------|------|
| `createTempDir()` | `() => Promise<string>` | 创建临时工作目录，返回路径 |
| `cleanupTempDir(dir)` | `(dir: string) => Promise<void>` | 删除临时目录 |
| `getLocalDirName(gitDir, localDir?)` | `(string, string?) => string` | 省略 local-dir 时，取 git-dir 路径最后一段 |
| `parseGitUrl(url)` | `(url: string) => {owner, repo}` | 解析 HTTPS / SSH URL |
| `validateUrl(url)` | `(url: string) => boolean` | URL 格式校验 |
| `promptOverwrite(dir)` | `(dir: string) => Promise<boolean>` | 交互询问是否覆盖目录 |
| `ensureOutputDir(dir)` | `(dir: string) => Promise<void>` | 检查目标目录是否存在且非空，存在则抛 DirExistsError |
| `setupAbortController(timeout)` | `(ms: number) => AbortController` | 创建全局 3 分钟超时控制器 |
| `handleInterrupt(tempDir)` | `(tempDir: string) => void` | 注册 SIGINT 处理器，清理后退出的 |

**→ 提交：** `git commit -m "add utils.ts"`

### 2.4 `src/cli.ts` — 命令行参数解析

**方案选择：** 使用 **Commander.js** 进行参数解析。

**职责：** 解析命令行参数、校验合法性、生成使用帮助。

```typescript
// 核心接口
interface CliOptions {
  gitUrl: string;
  gitDir: string;
  localDir?: string;
  branch: string;        // 默认 'main'
  quiet: boolean;        // 默认 false
}

function parseArgs(argv: string[]): CliOptions;
```

**校验逻辑：**
- `<git-url>` 必须符合 URL 格式（HTTPS / SSH 正则校验）
- `<git-dir>` 不能为空字符串
- `--branch` 值为有效的 Git 分支名（非空字符串）
- `--quiet` 为 boolean flag
- 参数数量不足时打印帮助信息并退出码 1

**→ 提交：** `git commit -m "add cli.ts"`

### 2.5 `src/clone.ts` — Git Sparse-Checkout 核心逻辑

**职责：** 封装 Git 命令执行，实现稀疏检出流程。

```typescript
// 核心接口
interface CloneOptions {
  gitUrl: string;
  gitDir: string;       // 相对于仓库根目录
  localDir: string;
  branch: string;
  quiet: boolean;
  signal?: AbortSignal; // 全局超时信号
}

async function sparseClone(options: CloneOptions): Promise<void>;
```

**内部流程函数：**

| 函数 | 调用 | 说明 |
|------|------|------|
| `buildCloneCmd(url, workDir)` | `sparseClone` | 构造 `git clone --filter=blob:none --no-checkout <url> <work-dir>` |
| `buildSparseCheckoutCmd(workDir, gitDir)` | `sparseClone` | 构造 `git -C <work-dir> sparse-checkout init --cone && git -C <work-dir> sparse-checkout set <git-dir>` |
| `buildCheckoutCmd(workDir, branch)` | `sparseClone` | 构造 `git -C <work-dir> checkout <branch>` |
| `copyOutput(workDir, gitDir, localDir)` | `sparseClone` | 将 `<work-dir>/<git-dir>` 复制到 `<local-dir>` |
| `validateGitVersion()` | `sparseClone` | 检查 Git 版本 >= 2.25.0 |

**→ 提交：** `git commit -m "add clone.ts"`

### 2.6 `src/index.ts` — 入口文件

**职责：** 整体流程调度，串联参数解析 → 校验 → 克隆 → 清理。

```
伪代码流程：

1. 解析参数 → CliOptions
2. 校验 Git 可用性
3. 检查本地目录（目标已存在 → 交互询问）
4. 注册 SIGINT 处理
5. 创建临时工作目录
6. 创建全局 AbortController（3 分钟超时）
7. 执行 sparseClone（传递 abort signal）
8. 清理临时目录
9. 输出 "save at <path>"
10. 退出码 0
```

**→ 提交：** `git commit -m "add index.ts"`

## Phase 5：边界情况处理验证清单

在实现过程中逐项确认以下场景已覆盖：

- [ ] Git 版本 < 2.25 → 友好提示
- [ ] 远程 URL 不可达 → 透传 Git 错误
- [ ] 指定目录在仓库中不存在 → 提示后退出码 1
- [ ] 本地目录已存在且非空 → 交互提示覆盖确认
- [ ] 用户输入无效覆盖回应 → 重复提示
- [ ] 全局 3 分钟超时 → AbortController 取消并清理
- [ ] Ctrl+C 中断 → 清理临时目录后退出码 130
- [ ] SSH URL 兼容 → 透传给 Git
- [ ] 磁盘空间不足 → 捕获异常并提示
- [ ] `git-dir` 为 `./` 或 `.` → 直接复制检出内容

**→ 提交：** `git commit -m "verify edge cases"`

```

                                                      │
                                                      ▼
                                              Phase3(每文件一提交)

                                              Phase4(构建+发布)
```

**提交规则：**
- 每个源文件/测试文件创建后立即提交一次
