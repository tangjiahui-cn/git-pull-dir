# git-pull-dir — 功能规格说明书

## 1. 概述

**git-pull-dir** 是一个命令行 CLI 工具，利用 Git 的 `sparse-checkout` 功能，从远程 Git 仓库中仅拉取指定的目录或文件到本地，避免下载整个仓库。

### 1.1 动机

大型 monorepo（如 `facebook/react-native`、`vercel/next.js`、`google/leveldb` 等）体积巨大，但开发者往往只关心其中某个子目录（如 `packages/core`、`examples/hello-world`）。传统 `git clone` 会下载整个仓库历史，浪费带宽和时间。本工具通过 sparse-checkout 实现部分克隆。

### 1.2 名称约定

- 包名：`git-pull-dir`（以 `git-` 为前缀，Git 别名机制使 `git pull-dir` 也可直接调用）
- 发布平台：npm（Public 包）
- 最低 Node.js 版本：v18.0.0（LTS）

## 2. 命令用法

```bash
git-pull-dir <git-url> <git-dir> [local-dir] [--branch=<name>] [--quiet]
```

或：

```bash
git pull-dir <git-url> <git-dir> [local-dir] [--branch=<name>] [--quiet]
```

### 2.1 参数说明

| 参数 | 是否必填 | 描述 | 示例 |
|------|----------|------|------|
| `<git-url>` | 是 | 远程 Git 仓库 URL（支持 HTTPS / SSH） | `https://github.com/facebook/react-native.git` |
| `<git-dir>` | 是 | 需要拉取的仓库内路径（支持目录或文件，相对于仓库根目录） | `packages/core` |
| `[local-dir]` | 否 | 本地存放目录。省略时取 `<git-dir>` 路径的最后一段作为目录名（如 `packages/core` → 本地目录为 `core`），放置在当前工作目录下 | `./my-core` |
| `[--branch=<name>]` | 否 | 指定拉取的分支名称，默认 `main` | `--branch=main` |
| `[--quiet]` | 否 | 静默模式，不显示步骤详情，仅显示 spinner | `--quiet` |

### 2.2 使用示例

```bash
# 基本用法：拉取 react-native 的 packages/core 到 ./core
git-pull-dir https://github.com/facebook/react-native.git packages/core

# 指定本地目录名
git-pull-dir https://github.com/facebook/react-native.git packages/core ./my-core

# 通过 git 子命令调用
git pull-dir https://github.com/vercel/next.js.git packages/next ./next-local

# SSH 地址
git-pull-dir git@github.com:facebook/react-native.git packages/core

# 指定分支
git-pull-dir https://github.com/facebook/react-native.git packages/core --branch=main
```

## 3. 内部实现原理

### 3.1 核心流程

```
输入参数
  │
  ├─→ [步骤1] 校验参数合法性（URL 格式、目录名合法性）
  │
  ├─→ [步骤2] 创建临时 clone（全局 3 分钟超时）
  │       git clone --filter=blob:none --no-checkout <url> <work-dir>
  │
  ├─→ [步骤3] 启用 sparse-checkout 并指定目录
  │       git -C <work-dir> sparse-checkout init --cone
  │       git -C <work-dir> sparse-checkout set <git-dir>
  │
  ├─→ [步骤4] 检出文件到目标目录，仅拉取指定目录内容
  │       git -C <work-dir> checkout <branch>（默认 main）
  │       将 <work-dir>/<git-dir> 复制到 <local-dir>
  │       （若 <git-dir> 为 ./ 或 .，则直接复制整个检出内容，不做路径拼接）
  │
  └─→ [步骤5] 清理临时工作目录
```

### 3.2 Git 技术选型

采用 **partial clone** (`--filter=blob:none`) + **sparse-checkout** 组合方案：

| 技术 | 作用 |
|------|------|
| `--filter=blob:none` | 初始克隆时不下载文件内容（blob），只下载 commit 树结构。文件在 checkout 时按需下载 |
| `--no-checkout` | 克隆后不立即检出工作区文件 |
| `sparse-checkout init --cone` | 启用锥形模式 sparse-checkout，仅允许显式声明的目录被检出 |
| `sparse-checkout set <dir>` | 声明仅需要检出的目录 |

这样不会下载仓库所有历史文件，只有指定目录的文件会实际传输。

## 5. 行为规范

### 5.1 成功场景

- 输出指定目录已成功同步到本地的信息
- 退出码 `0`

### 5.2 错误处理

| 场景 | 行为 | 退出码 |
|------|------|--------|
| 缺少必填参数 | 打印帮助信息 + 错误提示 | 1 |
| Git 未安装或版本过低（< 2.25） | 提示用户安装/升级 Git | 1 |
| 远程仓库 URL 无法访问 | 打印详细错误 | 1 |
| 指定目录在仓库中不存在 | 提示"目录不存在" | 1 |
| 本地目录已存在且非空 | 交互提示"目录已存在，是否覆盖？(yes/no)"，选 yes 则删除后覆盖，选 no 则取消拉取 | 0（取消时） |
| 网络超时 / 认证失败 | 透传 Git 原始错误信息 | 1 |
| 磁盘空间不足 | 捕获系统错误并提示 | 1 |
| **整个流程超过 3 分钟** | **终止进程，清理临时目录，提示"拉取超时"（使用全局 AbortController 控制总耗时，传递给每次 execa 调用的 cancel 信号，而非依赖单进程 timeout）** | **1** |

### 5.3 目录覆盖确认

- 当本地目录已存在且非空时，暂停拉取，向终端输出交互提示：

  ```
  目录 /Users/me/my-core 已存在，是否覆盖？(yes/no)
  ```

- 用户输入 `yes`（不区分大小写）→ 删除该目录，继续拉取
- 用户输入 `no`（不区分大小写）→ 取消拉取，退出码 0
- 用户输入其他内容 → 重复提示，直到输入 yes 或 no

### 5.4 中断处理

- 用户按 `Ctrl+C` 中断时，清理临时工作目录，然后退出进程，退出码 130
- 在退出前确保不会残留临时文件

### 5.5 进度反馈

- **默认模式**：逐行显示拉取步骤详情（克隆中 → 配置 sparse-checkout → 检出中 → 完成）
- **`--quiet` 模式**：不显示步骤详情，仅显示 `⠋` 样式的旋转动画（spinner）表示下载中
- 无论哪种模式，成功时都会输出一行 `save at <本地目录|本地文件>`
- 目录覆盖确认提示不受 `--quiet` 影响，始终显示

## 6. 输出示例

```bash
# 默认模式：显示步骤详情
$ git-pull-dir https://github.com/facebook/react-native.git packages/core ./my-core
clone in...
setting sparse-checkout...
checkout...
save at /Users/me/my-core

# --quiet 模式：仅显示 spinner
$ git-pull-dir --quiet https://github.com/facebook/react-native.git packages/core ./my-core
⠋ downloading...
save at /Users/me/my-core

# 目录已存在，用户选择覆盖（默认模式）
$ git-pull-dir https://github.com/facebook/react-native.git packages/core ./my-core
目录 /Users/me/my-core 已存在，是否覆盖？(yes/no)
yes
clone in...
setting sparse-checkout...
checkout...
save at /Users/me/my-core

# 目录已存在，用户选择取消
$ git-pull-dir https://github.com/facebook/react-native.git packages/core ./my-core
目录 /Users/me/my-core 已存在，是否覆盖？(yes/no)
no
cancelled
```

- 默认模式：逐行输出步骤文字 `clone in...` → `setting sparse-checkout...` → `checkout...` → `save at <path>`
- `--quiet` 模式：显示 `⠋` 样式的旋转动画（spinner），文字固定为 `downloading...`，完成后输出 `save at <path>`
- 步骤文字不带状态图标，仅为纯文字描述
- 取消时输出 `cancelled`，退出码 0
- 错误时输出错误信息，退出码非 0

## 7. 依赖

### 运行时依赖

| 包名 | 用途 | 理由 |
|------|------|------|
| `commander` | CLI 参数解析 | 成熟的参数解析库，支持子命令、自动生成 help |
| `execa` | 执行 Git 命令 | 比 `child_process.exec` 更好的错误处理、类型安全 |

推荐使用 **Commander.js** 而非自建参数解析，因为：
- 自动生成 `--help`、`--version`
- 参数校验（required、choices、validator）开箱即用
- 社区广泛采用，文档完善

如果追求最小依赖，也可以手写参数解析（本项目总代码量小，手写亦可）。

## 9. 边界情况与注意事项

### 9.1 Git 版本要求

- **最低 Git 版本：2.25.0**（`sparse-checkout` 命令在该版本引入）
- 启动时检测 Git 版本，低于 2.25 时提示升级

### 9.2 认证

- 不内建认证逻辑，完全依赖 Git 的已有认证机制：
  - SSH：`~/.ssh/id_rsa` 等
  - HTTPS：credential helper、git credential store
  - GitHub CLI：`gh auth`

### 9.3 并发

- 多个工具实例同时运行不会冲突（使用独立临时目录）
- 但拉取同一仓库到同一本地目录，后者会失败（目录已存在）

### 9.4 Windows 兼容性

- 路径处理使用 Node.js `path` 模块（自动处理 `\` 与 `/`）
- 核心流程基于 Git，而 Git for Windows 支持 sparse-checkout，可正常工作

### 9.5 大仓库 / 大目录

- Sparse-checkout + partial clone 组合确保只下载所需文件
- 指定目录内容依然可能很大（如几十 GB），Git 会按需流式下载，不会将仓库全部下载到本地
- 建议用户自行保证磁盘空间充足
- 默认模式下跳过 Git 原始进度输出，使用工具自身的步骤文字代替；`--quiet` 模式下用 spinner 替代
